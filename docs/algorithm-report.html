1What this component is

A drag-and-drop workflow editor (a mini Figma / JointJS canvas) rendered with D3.

A workflow is a graph of cards (nodes) joined by right-angle wires (links). There are three node types: root (the trigger), stage, and transition.

  • Cards are HTML, positioned with CSS.
  • Wires, grid, arrows and the minimap are SVG, drawn by D3.
  • Both layers share one zoom/pan transform, so they stay perfectly aligned.

Nearly every interaction runs the same loop:

// the core update cycle
change _nodes / _links  →  _autoLayout()  →  _renderLinks()

The genuinely hard engineering lives in two places: where cards go (layout, §3) and how wires dodge them (routing, §4). Everything else is plumbing.

3Auto-layout — "tidy tree"_autoLayout

Place cards in top-to-bottom layers, children fanned out under their parent, no overlaps.

Simple version: it's a family-tree layout. Work out how wide each branch needs to be, then center each parent over its children.

  1. Build parent → children from the links. Cards with no parent are roots.
  2. Measure subtree widths (bottom-up). A leaf is one card wide. A parent is as wide as its children combined, plus gaps. A visited set stops cycles from looping forever.
  3. Assign a depth (layer) to each card; remember the tallest card per layer.
  4. Compute each layer's Y. Layers with more wires passing through them get a larger gap.
  5. Place cards (top-down): position the parent, then spread its children symmetrically around its center using the widths from step 2.
The one feature to remember — pinning. When a user drags a card it becomes pinned. Auto-layout then leaves it exactly where the user put it, but still fans its children from that spot. The toolbar "Auto Layout" button (_autoLayoutAction) clears every pin to fully re-tidy — except nodes that take part in a cycle.
Cycles survive Auto Layout. Before re-tidying, _autoLayoutAction runs cycle detection and re-pins every node touched by a back-edge (both endpoints). Those nodes keep their current position so a hand-arranged loop is never disturbed; the rest of the graph fans out around them. This is why dragging a node into a cycle and then hitting Auto Layout leaves that node exactly where it was.

Why the two-pass design. Widths must be known before placement, because a parent is centered over the combined width of its children — which depends on grandchildren, and so on. So pass 1 (computeWidth) is strictly bottom-up (post-order), pass 2 (positionSubtree) is strictly top-down (pre-order). This is the classic Reingold–Tilford "tidy tree" split.

Complexity. Each node is visited once per pass, each link once when building the maps: time O(V + E)space O(V). The visited/positioned sets make it safe on cyclic graphs.

The exact width and Y formulas are also collected in the math cheat-sheet.

4Edge routing — the centerpiece

Draw a wire that is right-angled, never crosses a card, leaves/enters cards perpendicularly, and looks clean.

There is a primary router (A*) and several hand-written fallbacks for when A* can't find a path or the graph is too large.

4a · A* on a "Hanan grid"routeOrthogonal

The problem: the canvas is 6000×6000. Searching pixel-by-pixel is 36 million cells — hopeless.

The key trick (Hanan grid): a right-angle path only ever needs to turn at lines that line up with something important — a port or a card edge. Collect just those x-lines and y-lines; their grid of intersections is the search space. That's a few hundred points instead of 36 million, and the optimal path is still reachable.

How it works, in order

  1. Stubs. Push a short fixed segment straight out of each port (along the way it faces). The search runs between the stubs, forcing clean perpendicular exits. The true port points are stitched back on at the end.
  2. Grid lines. Collect x/y from ports, stubs and padded card edges, plus an outer "escape ring" so a wire can always go around everything. Sort, drop near-duplicates.
  3. Cost map. Each step costs length × region: open = ×1, padding band = ×6 (allowed but discouraged so wires don't hug edges), inside a card = ∞ (forbidden). Plus a flat bend penalty on every 90° turn.
  4. A* search. Manhattan heuristic. Two subtleties below.
  5. Rebuild & simplify to just the corners.
Why the state includes direction. The search state is (cell, arrival-direction), not just the cell — that's the only way to charge a bend correctly, because "did we turn?" depends on how we arrived. The search also starts already pointing in the port's direction, so going straight out is free and an immediate turn costs a bend.
Why a single midpoint sample is exact. A* only checks cost at the midpoint of each step — normally unreliable. It works here because every card edge is itself a grid line, so no boundary can fall between two adjacent lines. Each step lies entirely in one region, so one sample tells the truth.
Safety valve. If the grid would exceed maxGridCells (300k) A* returns null and the caller uses a fallback router.

The cost of one cell

Note the strict inequalities: the band's outer edge lines (obstacle ± padding) stay cheap, so a wire is allowed to sit exactly at the padding distance without being pushed all the way out to the margin.

The A* main loop

Why A* is guaranteed optimal here. A* returns the least-cost path whenever its heuristic h is admissible — never larger than the true remaining cost. Here the true remaining cost is at least the straight Manhattan distance (every step costs length × region with region ≥ 1, and bends/padding only add). The heuristic h = |Δx| + |Δy| equals that lower bound, so h ≤ true cost always ⇒ admissible ⇒ optimal. Folding direction into the state is what lets the bend penalty be part of g without breaking this.

Complexity. With N cards the grid has W ≈ 2N+c x-lines and H ≈ 2N+c y-lines. Each cell has 4 direction-states and 4 moves: states O(W·H) = O(N²)heap ops O(N² log N). The hard cap W·H > 300000 bounds this (~75 cards) before falling back.

The open set — a hand-rolled binary heap

There is deliberately no decrease-key: when a cheaper path to a state is found, a fresh entry is pushed and the stale one is skipped on pop via the closed set (the continue in the loop above). Simpler, and fast enough at this scale.

5Where wires attach + lane spreading

Before routing, decide which card edge each wire uses, and spread multiple wires so they don't pile up.

Choosing the side · _classifyLinkSides

  • Target clearly below → exit bottom, enter top (normal tree edge).
  • Target above → back-edge: hand off to _chooseBackEdgeSides, which tries a set of candidate side-pairs spanning all four edges (top, bottom, left, right for both ends), routes each with A*, and keeps the cheapest.
  • Cards side by side → connect on the facing sides.
Wires can attach to any side. The candidate list in _chooseBackEdgeSides isn't limited to left/right wraps — it includes top→bottom, top→left/right and left/right→bottom. So a cycle whose target sits above its source can leave the top of the source and enter the bottom of the target when that's the shortest clean path, instead of always wrapping around the sides. A* scores every candidate and the cheapest route wins, so the source dot and target arrow each land on whichever edge reads best.

Spreading along one edge · _spreadEdgePorts

Every wire on one edge — incoming and outgoing — is spread evenly, sorted by where its other end sits, which guarantees they don't cross right at the card.

Sibling "fan" lanes · _assignLanes

Children are split into those left and right of the parent. On each side the furthest child takes the top lane and nearer children tuck underneath — the clean symmetric fan, no crossings.

The lane number becomes a Y position in _forwardPolyline:

Why N+1 slots, not N. We want N lanes with equal gaps and a gap before the first and after the last (so no wire hugs a card edge). Placing lane i at fraction t = (i+1)/(N+1) for i = 0…N−1 yields positions 1/(N+1), 2/(N+1), … N/(N+1) — exactly N+1 equal intervals across the band, with the endpoints left empty. Dividing by N−1 instead would push the outermost wires onto the edges.

Nested cycle lanes · _assignBackEdgeLanes

Cycles on the same side are sorted by span; the widest wraps furthest out, so nested loops form concentric arcs instead of crossing.

6Cycle detection_computeBackEdges

Find links that loop back to an ancestor, so they can be styled (amber, dashed).

How: a depth-first search tracking which nodes are currently on the recursion stack. A link to a node already on the stack is a back-edge.

The key distinction: an edge to a node in onStack is a true cycle; an edge to a node in done (already finished, but not an ancestor) is a harmless forward/cross edge and is not flagged. time O(V + E)

Topology, not geometry. This is based on graph structure, not screen position — so dragging a card physically above its neighbor will not make a normal edge look like a cycle.

Reused by Auto Layout. The same back-edge set drives two other features: the amber/dashed styling of cycle wires, and — via _autoLayoutAction — protecting cycle nodes from being re-flowed (their endpoints are re-pinned before the tidy layout runs).

7Line jumps_assignLineJumps

Where a later wire crosses an earlier one, draw a small arc "hop" so the crossing reads clearly.

Every horizontal segment of a later wire is tested against every vertical segment of earlier wires (and vice-versa). Because all segments are axis-aligned, the intersection test is a couple of range checks — no general line-line math needed.

Only the later wire (index j > i) hops, so each crossing is bridged exactly once. _lineWithHops then turns each recorded scalar into a small SVG arc (A r,r 0 0,sweep …) while drawing that segment, merging hops that are closer than one arc-diameter so they don't overlap.

Cost. It compares every segment pair across all wires: O((Σ segments)²). Fine for workflow-sized graphs (tens of wires, a few segments each); if graphs ever get huge this is the first place to add a sweep-line or spatial index.

8Smaller algorithms

Feature Function Idea
Drop a card without overlaps _findFreePosition Spiral outward ring by ring; drop into nearest free slot. Existing cards never move. Takes an optional excludeIds set so a node isn't counted as overlapping itself.
Nudge a freshly placed card clear _resolveOverlaps After a new card is (auto-)placed, if it still overlaps another node it slides to the nearest free slot (via _findFreePosition) and is pinned. See the note below.
Connect-drag → nearest card _hitTestNode, _findClosestNode Bounding-box / distance checks.
Add / delete a card (animated) CSS node-enter / node-exit, _deleteNode New cards fade+scale in on insert; deletes flag the card, play the exit animation, then remove it after NODE_EXIT_MS. See the note below.
Undo / redo _pushUndo Deep-clone the whole {nodes, links} state onto two stacks (capped 50).
Fit to screen _fitToScreen Bounding box of all cards → scale + offset that centers it.
Minimap _updateMinimap Same graph drawn small, with a viewport rectangle.
Execution animation _runExecutionAnimation BFS from roots → visit order; a timer lights up each node and wire (cycle links last).

Segment-vs-rectangle test (Liang–Barsky) · _segIntersectsRect

Used by the fallback routers to ask "does this candidate segment hit a card?". It parameterizes the segment as P(t) = A + t·(B−A) for t ∈ [0,1] and clips that interval against the four rectangle edges; if the surviving interval is non-empty, they intersect.

Collision-free placement (spiral ring search) · _findFreePosition

To drop a new card without nudging any existing one, scan outward in square "rings" from the desired spot and return the first free slot.

Why a new card no longer overlaps after dragging. Two add-paths feed this: dropping from the palette (_addNodeFromPalette) now routes the drop point through _findFreePosition before placing the card; and the "+"/Create-New-Stage path (_addNodeBelow) runs the normal tidy layout, then calls _resolveOverlaps([transition, stage]) to slide any new card clear of a pinned (dragged) neighbour the layout couldn't move. The excludeIds argument is what lets a card ignore itself while testing — without it a node already in _nodes would always report a self-collision.

Add / delete animations

Edits don't snap — they animate, JointJS-style. There are two cooperating pieces: a per-card scale/fade (CSS, on the child workflowNode component) and a whole-graph position glide (JS, _animatePositions) that moves every card to its new spot while the wires follow.

  • Enter (scale/fade) — the .node-card always carries a node-enter keyframe (fade + scale-up with a gentle overshoot). Because LWC reuses DOM elements by their key, it only fires when a card is first inserted (a genuinely new node, or initial load) — never on re-render or during a drag.
  • Exit (scale/fade) — deletion can't be instant or there'd be nothing to animate. _deleteNode flags the doomed card(s) into the tracked _deletingNodeIds set, which surfaces as is-deleting → the node-card--deleting class → the node-exit keyframe (fade + scale-down). Only after NODE_EXIT_MS (220 ms, matched to the CSS) does a deferred callback drop the nodes/links and then glide the survivors in to close the gap.
The glide tween — _animatePositions(computeTargets, onDone). This is what turns a layout change into motion instead of a jump. It (1) snapshots every card's current position, (2) runs computeTargets() — usually _autoLayout() — to work out the new positions, then (3) tweens every card from old→new over ~420 ms (ease-out-cubic), re-routing the wires on every frame so they stay attached to the moving cards. New cards have no "old" position, so they appear at their target and let the CSS enter animation play; a freshly added stage is first seeded at its parent's position, so it looks like it grows out of the parent. Used by add (_addNodeBelow), delete, collapse/expand, and the Auto Layout button. A drag cancels any running tween (_layoutAnimRaf) so the pointer takes over instantly.
Connectors don't flip mid-glide. If the wires were re-classified from the live positions every frame, a card passing above/beside another would make its connector jump between edges and then snap straight. To prevent that, the tween publishes each node's final position in _sideRefPos; the side/lane decisions (_classifyLinkSides, _chooseBackEdgeSides, port spread & lane ordering) read positions through _refX/_refY, which return that settled position while animating. So a wire keeps the edge it will ultimately attach to for the whole glide and only its shape eases into place; endpoint coordinates still track the live cards. New nodes are also seeded just below their parent so the edge is a clean forward (bottom→top) one from the first frame.
Duration coupling. The NODE_EXIT_MS constant in the TS and the node-exit animation duration in workflowNode.css must stay in sync — if the CSS is slower than the timer the card vanishes mid-animation; if it's faster the card sits frozen before removal.

Execution order (BFS) · _runExecutionAnimation

The animation walks the graph breadth-first from the roots, recording the order nodes/links are first reached. Edges that lead to an already-visited node are cycles and are appended to play last.

9Math cheat-sheet

The actual formulas, lifted from the code. Symbols: k = number of children, d = layer depth.

Subtree width (bottom-up recurrence)

// leaf:   W(n) = NODE_W
// internal node with children c₁…ck:
W(n) = max( NODE_W ,   Σi=1..k W(cᵢ)  +  (k − 1) · MIN_SIBLING_GAP_X )

Layer vertical position

Y(0) = 200
Y(d+1) = Y(d) + maxH(d) + BASE_LAYER_GAP + max(0, crossings(d) − 1) · LANE_SPACING
// crossings(d) = links that span across the gap below layer d

Child placement (fan around parent center)

startX = centerX − totalChildW / 2
childCenterᵢ = startX + (Σj<i W(cⱼ) + (i)·GAP) + W(cᵢ)/2

A* cost & heuristic

region(p) = ∞ inside a card  |  6 in padding band  |  1 in open space
step  = length · region(midpoint)  +  (turned ? bendPenalty : 0)
g(n)   = g(prev) + step
h(n)   = |xₙ − x_goal| + |yₙ − y_goal|   (Manhattan)
f(n)   = g(n) + h(n)
// h never overestimates (bends & padding only add cost) ⇒ admissible ⇒ optimal
// state key packs cell + direction:
stateKey(i, j, dir) = (j · W + i) · 4 + dir

Back-edge side selection

cost(side-pair) = routeLength + bends · BACKEDGE_BEND_COST
// pick the lowest-cost side-pair across the candidates, which span all four
// edges: top→bottom, top→left/right, left↔right, left/right→bottom, left/left, right/right

Lane → coordinate (sibling fans & sideways wires)

t = (laneIndex + 1) / (laneCount + 1)   // divide band into N+1 slots
midY = stubTop + (stubBot − stubTop) · t
// the +1 keeps the first/last wire off the card edges

Nested cycle offset

routeOffsetᵢ = i · BACKEDGE_LANE_GAP   // sorted by span; widest cycle = largest i = outermost

Fit to screen

scale = min( viewW / contentW , viewH / contentH , 1.2 )
tx = viewW/2 − centerX · scale    ty = viewH/2 − centerY · scale

Screen ↔ canvas conversion (drag math)

canvasX = (clientX − containerLeft − t.x) / t.k
canvasY = (clientY − containerTop − t.y) / t.k
// t = current zoom transform; t.k = scale, (t.x,t.y) = pan

Hanan grid size & the A* state space

W = |unique xs| ≤ 4 + 2·|obstacles| + 2   // 4 endpoints/stubs, 2 padded edges each, 2 ring lines
H = |unique ys| ≤ 4 + 2·|obstacles| + 2
states = W · H · 4 directions    bail if   W · H > maxGridCells (300000)
// with N cards: W,H = O(N) ⇒ states = O(N²) ⇒ A* runs in O(N² log N)

Even port spread along an edge

// n members on one edge, parameter for the i-th (0-based):
coordᵢ = lo + (hi − lo) · ( i / (n − 1) )   for n > 1
coord₀ = (lo + hi) / 2   for n = 1
// horizontal edge: lo/hi inset by PORT_EDGE_MARGIN; vertical: by SIDE_VERTICAL_OVERLAP

Line-jump arc & hop merging

arc at scalar h:   L(h − r·dir)  →  A r,r 0 0,sweep  →  (h + r·dir)   // r = HOP_RADIUS
keep next hop only if   |h − h_prev| > 2r + 2   // else they'd visually overlap

Complexity summary

Algorithm Time Space
Auto-layout (both passes) O(V + E) O(V)
A* route (per wire) O(N² log N) O(N²)
All wires routed O(E · N² log N) O(N²)
Cycle detection (DFS) O(V + E) O(V)
Line jumps O(S²) over all segments O(S)
Free-position search O(R² · V) (R = rings ≤ 12) O(R²)
Heap push / pop O(log n) O(n)

V = nodes, E = links, N = cards (obstacles), S = total wire segments.

10Constants & tuning

Constant Meaning
NODE_W Card width (all cards share it).
NODE_H_MAP / NODE_H_COLLAPSED Card heights by type, expanded vs collapsed.
BASE_LAYER_GAP, LANE_SPACING Vertical spacing between layers / extra per crossing.
MIN_SIBLING_GAP_X Minimum horizontal gap between sibling subtrees.
ROUTE_DEFAULTS A* tuning: padding, paddingPenalty, bendPenalty, portStub, margin, maxGridCells.
CORNER_RADIUS, HOP_RADIUS Corner rounding / line-jump arc size.
Tuning tips
  • Wires hugging cards → raise padding or paddingPenalty.
  • Too many zig-zags → raise bendPenalty.
  • A* falling back on big graphs → raise maxGridCells (watch perf).
  • Sibling wires overlapping → that's the lane logic (§5), not the router.
Mental model. Every hard problem here is solved by turning something continuous into something small and discrete: routing → A* over a few hundred grid points; wire spreading → divide a band into N+1 slots; placement → spiral ring search. Pipeline to remember: layout → ports → A* route → render.
